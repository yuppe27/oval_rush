# OVAL RUSH - ブラウザレーシングゲーム 仕様書

> Version 2.0 - 全面改訂版

## 1. ゲーム概要

| 項目 | 内容 |
|------|------|
| タイトル | **OVAL RUSH** (オーバルラッシュ) |
| ジャンル | アーケードスタイル 3Dレーシングゲーム |
| プラットフォーム | Webブラウザ (PC優先 / モバイル簡易対応) |
| 技術スタック | HTML5 / JavaScript (ES Modules) / Three.js / 自前簡易物理 |
| インスピレーション | 90年代アーケードレーシングゲーム全般 |

### 1.1 コンセプト

90年代アーケードレーシングの爽快感をブラウザ上で再現する。
ストックカーレースの興奮を、シンプルな操作と派手な演出で体験できるゲーム。
制限時間をチェックポイントで延長しながら上位入賞を目指すアーケードスタイル。

### 1.2 ゲームの核となる体験

1. **スピード感** - 時速300km超の体感速度をカメラワークとエフェクトで演出
2. **ドリフトの快感** - 簡単操作でド派手なドリフトが決まる
3. **接戦の緊張感** - ラバーバンドAIにより常に僅差のレース展開
4. **制限時間のスリル** - チェックポイント制による緊張と安堵の繰り返し

### 3.2 ドライビングモデル (アーケード簡易物理)

本ゲームではリアルな物理シミュレーションではなく、
爽快感を優先したアーケード簡易物理モデルを採用する。

```
■ 基本挙動
  - 加速: パラメータ「加速力」に基づく一定加速。速度が最高速に近づくと減衰
  - 減速: ブレーキは即座に効く (アーケード的な即応性)
  - 旋回: 速度に応じてステアリング感度が変化 (高速時は鈍化)
  - 摩擦: コース路面に固定の摩擦係数。砂利・芝ではグリップ大幅低下

■ ドリフト
  - 発動条件: 専用ボタンなし。以下の3条件をすべて満たすと自動発動
      1) 速度が最高速の30%以上
      2) ブレーキボタン押下後0.2秒以内である
      3) その0.2秒の間にステアリング入力量が78%以上まで入る
         ※ すでにハンドルを切っている状態でブレーキを押しても発動しない
  - 挙動: 後輪グリップが一時的に低下し、車体が横滑り
      - ドリフト中は速度が緩やかに減衰 (0.3%〜1.2%/秒、ドリフト品質に応じて変動)
      - スロットルオン中は減衰がさらに軽減される
      - スロットルオフ時は追加減衰 (3.8%/秒)
      - ドリフト品質に応じた駆動力 (20%〜52%) でコーナー中も速度維持しやすい
      - 出口で短いトラクション補助 (0.08〜0.18秒) が入る
  - ドリフト角度: ベース約20° + ステア量・カウンターステアに応じて変動
      - 理論最大: 約86° (ただしスピン閾値83°を超えるとスピン移行)
      - 品質評価の理想角度帯: 約15°〜41°
      - 過大角度 (39°以上): 追加速度ペナルティ発生
  - ドリフト品質: ドリフト角度と操舵の整合性で評価 (0.0〜1.0)
      - 品質が高いほど速度維持・旋回ボーナス・出口加速が向上
  - コース追従アシスト: ドリフト中は通常時より強い自動ヨー補正とセンター寄せ補正が入る
      - 曲率の強い区間でも自車がコース外へ膨らみにくい
  - スピン移行: ドリフト角83°超の状態が続くと不安定蓄積→スピン
  - 最大持続時間: 発動時の速度に応じて変動
      - 低速エントリーでは短め (約1.6秒)
      - 高速エントリーでは長め (最大約3.0秒)
  - ドリフト終了条件:
      - ドリフト角が約7°未満に縮小 + ステア入力なし
      - カウンターステアで姿勢が整った
      - 速度が最高速の16%未満に低下
      - そのドリフトに設定された持続時間を超過
  - ドリフトゲージメータ: 廃止。HUDには専用メータを表示しない
  - ブースト発動: ドリフト終了時、内部評価値 (持続時間 × 品質) が一定以上ならブースト
      - 短いドリフトでも低評価ブーストを得られる
      - 低評価: 0.22秒 / 中評価: 0.55秒 / 高評価: 1.0秒
      - 評価が低すぎる場合はブーストなし

■ ブースト
  - ブースト速度: 最高速の120% まで一時的に加速
  - ブースト時間はドリフト内部評価値 × ボーナス係数(0.45〜0.80)で決定
      - ボーナス係数 = 0.45 + ドリフト脱出アライメント×0.2 + 脱出時スロットル×0.15
  - 高評価: 基本1.0秒 × 係数 → 約0.45〜0.80秒
  - 中評価: 基本0.55秒 × 係数 → 約0.25〜0.44秒
  - 低評価: 基本0.22秒 × 係数 → 約0.10〜0.18秒
  - 評価不足: ブーストなし
  - ブースト加速力: 通常加速の65%

■ スリップストリーム
  - 発動条件: 前方車両の後方 3〜15m 以内、角度 ±10° 以内
  - 効果: 最高速の103% まで徐々に加速 (1.5秒で最大効果到達、解除時0.45秒で減衰)
  - 視覚エフェクト: 半透明の空気ライン表示

■ 衝突
  - 壁衝突 (浅い角度): 速度2〜6%減 + 0.05〜0.14秒間操作鈍化
  - 壁衝突 (中程度): 速度14〜34%減 + 0.16〜0.34秒間操作鈍化
  - 壁衝突 (正面衝突): 速度68%減 + スピン + 0.55秒間操作鈍化
  - 車両同士: 横方向に弾かれる (リタイア・破壊なし)
```

### 3.3 トランスミッション / パワートレイン

```
共通方針:
- 車両性能は「最高速」「基礎加速度」に加え、ギア比テーブル / 最終減速比 / シフトしきい値で定義する
- ギア比は加速倍率専用、最高速は車両固有の maxSpeed を基準とし、ギア上限はハードカットではなくソフトリミッタとして作用する
- 各ギアは speedRangeKmh(min/max) を持ち、その範囲から簡易 RPM を推定する
- 簡易 RPM は以下へ使用する:
  - トルクバンド補正
  - AT の自動変速判断
  - MT の過回転抑制

標準ギア構成:
- 4速
- 各段は以下を持つ:
  - ratio: その段の駆動力倍率
  - speedRangeKmh: その段で自然に伸びる速度帯
  - upshiftKmh: AT の基本シフトアップ速度
  - downshiftKmh: AT の基本シフトダウン速度

プレイヤー車挙動:
- スロットル加速は baseAcceleration × gearRatio × finalDrive × torqueBand で決定
- torqueBand は低回転で弱く、中速域で最も強く、高回転でやや落ちる
- 現在ギアの上限速度を超えた場合はソフトリミッタで駆動力を減衰させる
- シフトアップ時は短いトルクカットを入れる
- シフトダウン時は固定減速ではなく、回転差に応じた軽いエンジンブレーキを入れる
- boost / drift exit / slipstream は最高速上限だけでなく、駆動力への補正量も持つが、ギア段とトルクバンドを無視して直接速度を注入しない

AT:
- gearTable.length を段数として扱い、固定4速前提にしない
- throttle 量と現在速度を使って upshift/downshift を判断する
- 変速直後は短時間の再変速禁止時間を設け、ハンチングを防ぐ

MT:
- Z / X でシフトアップ / シフトダウン
- シフトダウン時は回転差に応じて速度をわずかに落とす
- 過回転域でアクセル全開を続けても、次段に上げない限り加速は頭打ちになる

AI車挙動:
- AI もプレイヤーと同じ vehicle preset を共有し、車種ごとの maxSpeed / acceleration / gearTable / finalDrive を使う
- AI の目標速度は waypoint 推奨速度、ラバーバンド、交通状況から決定する
- 実速度への追従は車種別加速力と簡易ギア制御で行う
- ドリフト中は速度ボーナスを付与せず、減速ペナルティを緩和する程度に留める
- 接触補正は「瞬間的な押し出し加速」ではなく、相対速度に応じた軽い減速と位置補正を優先する
```

## 4. コース設計

### 4.1 コースデータ形式

コースはスプライン曲線 (Catmull-Rom) で中心線を定義し、幅・バンク角・高度を制御点ごとに指定する。

```
コースデータ構造:
{
  name: "Thunder Oval Speedway",
  controlPoints: [
    { x, y, z, width, bankAngle },  // 各制御点
    ...
  ],
  checkpoints: [ pointIndex, ... ],  // チェックポイント位置 (制御点のインデックス)
  startLine: pointIndex,             // スタートライン位置
  sceneryObjects: [                  // 背景オブジェクト配置
    { type: "stadium", position, rotation, scale },
    ...
  ]
}
```

路面メッシュは制御点から自動生成する (押し出し + UV展開)。

### 4.5 コース別タイムバジェット検証

レースが適切な長さになるか検証する。

> **注意**: 最終ラップの最後のCP通過後はゴールが近いため延長不要。
> 有効延長回数 = (CP数 × 周回数) - 1 とする（ゴール直前のCPは延長せずそのままゴール判定）。

| コース | 想定ラップ | 総レース時間 | CP数×周回 | 有効延長回数 | 最大制限時間 |
|--------|-----------|-------------|-----------|-------------|-------------|
| Thunder Oval | ~25秒/周 | ~200秒 (3.3分) | 2×8=16 | 15回 | 40 + 15×12 = **220秒** |
| Seaside Grand | ~55秒/周 | ~220秒 (3.7分) | 4×4=16 | 15回 | 45 + 15×14 = **255秒** |
| Mountain Apex | ~100秒/周 | ~200秒 (3.3分) | 6×2=12 | 11回 | 50 + 11×16 = **226秒** |

- いずれも3〜4分程度のアーケードレースとして適切
- 最大制限時間は総レース時間を上回り、初心者でも完走可能な設計
- HARD難易度ではチェックポイント延長を-2秒し、上級者を追い込む
- Mountain Apex は余裕が少ないため、HARD時は延長 -1秒 に緩和することも検討

## 5. 車両設計

### 5.1 パラメータ定義

各値は 1〜10 のスケール。内部では実数値に変換して使用する。

| パラメータ | Falcon MK-I | Bolt RS | Ironclad GT | 内部変換 |
|-----------|-------------|---------|-------------|----------|
| 最高速度 | 7 | 10 | 5 | 260〜320 km/h |
| 加速力 | 7 | 5 | 8 | 加速度係数 |
| ハンドリング | 7 | 5 | 9 | ステアリング応答速度 |
| ドリフト性能 | 7 | 9 | 5 | ドリフト時のグリップ減少率 |
| 車体重量 | 6 | 4 | 9 | 衝突時の押し負けにくさ |

### 5.2 車両キャラクター

**Falcon MK-I (ファルコン)**
- コンセプト: 万能型。初心者から上級者まで扱いやすい
- 外観: クラシックなストックカーフォルム、丸みのあるボディ
- カラー: 赤 (デフォルト) / 青 / 白
- 推奨コース: 全コース

**Bolt RS (ボルト)**
- コンセプト: 最高速特化。直線番長だがコーナーは腕が問われる
- 外観: ロー&ワイドなウェッジシェイプ、大型リアウィング
- カラー: 黄 (デフォルト) / 黒 / 緑
- 推奨コース: Thunder Oval (長い直線を活かせる)

**Ironclad GT (アイアンクラッド)**
- コンセプト: グリップ型。コーナーに強く安定した走りが持ち味
- 外観: がっしりとした箱型ボディ、重厚感のあるデザイン
- カラー: オレンジ (デフォルト) / 紫 / シルバー

### 5.3 3Dモデル仕様

ブラウザ負荷を考慮し、ローポリゴンスタイルを採用する。

## 6. AI (ライバル車)

### 6.1 AI難易度

難易度はモード選択後に3段階から選択する。

| 設定 | AIレベル | AI最高速 | ラバーバンド強度 | CP延長補正 |
|------|---------|---------|-----------------|-----------|
| EASY | 1 | 車両最高速の85% | 強 (差を縮めやすい) | +0秒 |
| NORMAL | 2 | 車両最高速の95% | 中 | +0秒 |
| HARD | 3 | 車両最高速の100% | 弱 (差がつきやすい) | -2秒 |

### 6.2 AI走行ロジック

```
1. ウェイポイント追従
   - コース中心線のスプライン上に等間隔でウェイポイントを配置
   - 各AIは次のウェイポイントに向けてステアリングを計算
   - レーシングライン: イン側に寄る補正値を個体ごとにランダム化

2. 速度制御
   - 次ウェイポイントまでのカーブ角度を先読みし、ブレーキ量を決定
   - カーブ角度 > 30°: 軽くブレーキ
   - カーブ角度 > 60°: 強くブレーキ
   - 直線: 全開加速

3. 回避行動
   - 前方 20m以内に他車を検出 → 左右の空きスペースを判定し回避
   - 回避方向はレーシングラインに近い側を優先

4. ラバーバンド処理
   - プレイヤーとの距離を毎フレーム計測
   - プレイヤーが大幅リード時: AI速度を最大+10%補正
   - プレイヤーが大幅ビハインド時: AI速度を最大-8%補正
   - HARD難易度ではこの補正値を半減 (よりシビアなレース)

5. 個性パラメータ (AI車両ごとにランダム生成)
   - 攻撃性 (0.0〜1.0): 高いほど積極的に追い越す
   - 安定性 (0.0〜1.0): 高いほどミスが少ない
   - ライン精度 (0.0〜1.0): 高いほど理想ラインに近い走行
```

### 6.3 AI描画の最適化

```
全AI車両に物理演算を適用するのは重すぎるため、以下の簡易方式を採用:

- AI車両は物理エンジンを使用しない
- ウェイポイント上の位置を毎フレーム更新するだけの「レール走行」
- プレイヤー車両との衝突判定のみ、バウンディングボックスで簡易計算
- AI同士の衝突は判定しない (重なりは許容、見た目上の回避行動で緩和)
- 画面外のAI車両は描画スキップ (位置計算のみ継続)
```

### 9.2 物理演算の方針

```
■ Rapier (WASM) を不採用とする理由:
  - 本ゲームは「アーケード簡易物理」であり、リアルな剛体シミュレーションは不要
  - Rapier WASMバイナリは約2MB。初回ロード3秒以内の目標に対してリスクが高い
  - ドリフト/ブースト等の挙動はパラメータ制御であり、物理エンジンと相性が悪い
  - プレイヤー車両1台のためだけに物理エンジンを導入するのはオーバーヘッド

■ 自前簡易物理の方針:
  - 加速/減速: パラメータベースの速度増減
  - 旋回: 速度に応じたステアリング感度で角度を直接制御
  - ドリフト: フラグベースでグリップ係数を切替、横滑り角度をパラメータ制御
  - 壁衝突: コース中心線からの距離 + コース幅で壁判定。反射ベクトルで跳ね返り
  - 車両衝突: AABB (軸平行バウンディングボックス) で簡易判定
  - AI車両はウェイポイントベースのレール走行 (物理不使用)
  - AI同士の衝突は判定しない (見た目上の回避行動で緩和)

■ 将来的にリアル物理が必要になった場合:
  - PlayerVehicle.js の物理計算部分を差し替えるだけで Rapier 等に移行可能な設計とする
```

### 9.4 パフォーマンス目標と最適化戦略

| 項目 | PC目標 | モバイル目標 |
|------|--------|-------------|
| フレームレート | 60 FPS | 30 FPS |
| 初回ロード | 3秒以内 | 5秒以内 |
| メモリ使用量 | 256 MB以下 | 128 MB以下 |

```
最適化戦略:
  1. ローポリモデル: 車両 500-800 tri、背景物 200-500 tri
  2. テクスチャアトラス: 車両は256×256、背景は512×512に統合
  3. インスタンス描画: 同一モデルのAI車両はInstancedMeshで一括描画
  4. 視錐台カリング: Three.js標準のfrustum cullingを有効化
  5. 背景LOD: 遠方の背景オブジェクトは板ポリ(ビルボード)に差替
  6. AI簡易処理: 物理演算なし、AIはレール走行
  7. パーティクル制限: 同時表示パーティクル数を200個 (モバイル100個) に制限
  8. モバイル時: シャドウ無効、ポストエフェクト無効、パーティクル半減
  9. ポストエフェクト制御:
     - PC High: ブルーム + 放射状ブラー (ブースト時のみ) + FXAA
     - PC Low: FXAA のみ
     - モバイル: ポストエフェクト全無効
     ※ 放射状ブラーとブルームの常時併用はフルスクリーンパス2回分の負荷があるため、
       ブースト中の短時間のみ放射状ブラーを有効化する
  10. ドローコール削減:
     - AI車両は車種ごとにInstancedMeshで一括描画 (最大3ドローコール)
     - スキッドマークはデカールではなくライン描画で代用 (動的テクスチャ不要)

ライバル台数の調整 (パフォーマンス対策):
  - Thunder Oval: 15台 → 11台 (自車含め12台) に削減
    理由: 16台同時描画 + タイヤスモーク等は中〜低スペックPCで60FPS維持が困難
  - Seaside Grand: 11台 → 9台 (自車含め10台) に削減
  - Mountain Apex: 7台 → 据え置き (元から少ない)
  ※ 実装後のプロファイリングで余裕があれば元の台数に戻す
```

### 9.6 ゲームループ設計

```javascript
// 概念的なゲームループ構造
const FIXED_TIMESTEP = 1 / 60;  // 物理演算は60Hz固定
let accumulator = 0;

function gameLoop(timestamp) {
    const deltaTime = (timestamp - lastTime) / 1000;
    lastTime = timestamp;
    accumulator += deltaTime;

    // 入力取得 (毎フレーム)
    inputManager.update();

    // 物理演算 (固定タイムステップ)
    while (accumulator >= FIXED_TIMESTEP) {
        physicsWorld.step(FIXED_TIMESTEP);
        playerVehicle.fixedUpdate(FIXED_TIMESTEP);
        aiController.update(FIXED_TIMESTEP);
        raceManager.update(FIXED_TIMESTEP);
        accumulator -= FIXED_TIMESTEP;
    }

    // 描画 (可変フレームレート)
    const alpha = accumulator / FIXED_TIMESTEP;
    renderer.render(alpha);  // 補間描画
    hud.update();

    requestAnimationFrame(gameLoop);
}
```

#### Phase 3 実装詳細仕様 (確定版)

以下は Phase 3 を「実装完了」と判定するための詳細仕様。

**A. スタートグリッド演出**

```
レース開始シーケンス:
  1) grid_intro (3.5秒)
     - AI車両をスタートライン後方へ 2列の千鳥配置で整列
     - プレイヤー操作を無効化
     - カメラは俯瞰→斜め前方→プレイヤー背後へ遷移するシネマティック移動
     - HUD中央に "ON THE GRID" を表示

  2) countdown (3, 2, 1)
     - 既存カウントダウン表示を使用
     - プレイヤー操作は引き続き無効

  3) GO!
     - レース状態を racing に遷移
     - プレイヤー入力とAI走行を同時に有効化
```

受け入れ条件:
- リトライ時にも同一シーケンスが必ず再生される
- grid_intro 中に車両が勝手に走り出さない
- カウントダウン開始時にはカメラが追従視点へ収束している

**B. ウェイポイントシステム**

```
Waypoint生成:
  - CourseBuilder.sampledPoints から一定間隔で waypoint を生成
  - 各 waypoint は以下を保持:
      id, sampleIndex, t, position, forward, width,
      curvature(先読み接線角), suggestedSpeed

  - suggestedSpeed は曲率ベースで算出:
      直線に近いほど高く、急カーブほど低い
      （例: maxSpeedの55%〜100%を範囲にする）

AI追従:
  - 各AIは currentWaypointIndex を持つ
  - 進行方向で waypoint を順送りし、目標速度を決定
  - 実際の速度は車種別 accel/brake と簡易ギア制御で滑らかに追従
  - ラバーバンド係数は最終目標速度に乗算
  - ラバーバンドは「不自然な最高速付与」ではなく、目標速度と加速余力の補正として使う
  - AIごとの車種差は maxSpeed だけでなく、低速加速と中速の伸びにも現れる

回避:
  - 近接AIを検出した場合、laneTarget を一時的に左右へオフセット
  - laneTarget はコース幅内でクランプする
  - 追突回避時は先行車速度へ急に吸着させず、相対速度に応じて減速する
```

受け入れ条件:
- AI更新が「ウェイポイント情報」を参照して速度決定している
- 直線で加速し、コーナー進入前に減速する挙動が確認できる
- 既存の InstancedMesh 描画方式を維持する

## 12. 著作権に関する注意事項 (まとめ)

| 対象 | 方針 |
|------|------|
| ゲームタイトル | 「OVAL RUSH」完全オリジナル。DAYTONA等の商標を含まない |
| コース名 | すべて架空名称。実在サーキット名を使用しない |
| 車両名 | すべて架空名称。実在車種名・メーカー名を使用しない |
| 車両デザイン | 「ストックカー風」の一般的デザイン。特定車種の模倣を避ける |
| コースレイアウト | 実在サーキットのトレースを避ける。オリジナル設計 |
| BGM/SE | オリジナル制作またはCC0/フリーライセンス素材のみ |
| フォント | SIL OFL / Apache License等のフリーフォントのみ |
| ゲームシステム | チェックポイント制限延長等は一般的メカニクス。特許回避済み |
| UI/演出 | 「アーケードレーシング」の一般的な表現に留める |
